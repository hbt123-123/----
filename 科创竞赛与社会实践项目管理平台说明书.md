# 中南民族大学资源与环境学院团委学生会创新实践部信息化管理平台

## 需求规格说明书

***

### 一、引言

#### 1.1 背景

资源与环境学院团委学生会创新实践部（简称“创实部”）负责学院的寒暑假社会实践、科研立项、“互联网+”和“挑战杯”竞赛的组织开展与院内考核，以及共青团重点调研课题申报等工作；同时负责大学生其他学科竞赛的通知传达、人员统计、费用收集等工作。目前主要存在以下问题：

- 比赛项目种类多，且多为多阶段长周期项目，进度跟踪混乱。
- 材料收集依赖人工线下传递，易丢失、易混淆。
- 收齐材料后需手动制作 Excel 汇总表，工作量大且容易出错。
- 副部长、干事任务分配依赖口头或聊天工具，缺乏系统记录和提醒，逾期频发。
- 部长难以全局掌握所有项目的实时进展和人员负荷。
- 竞赛通知传达、人员统计、费用收集等事务性工作散落于聊天工具，缺乏统一记录。

#### 1.2 建设目标

建立一个轻量级的信息化管理平台，以网站形式提供，后端可部署于轻量化服务器，满足约 13-15 人日常使用，实现：

- 比赛项目全生命周期（申报-中期-结题）的结构化管理。
- 材料收集、审核、汇总、报送的线上流水线，彻底告别手工表格。
- 部长、副部长、干事三级协同，权责清晰、进度可视。
- 支持多项目、多阶段并行，自动提醒与阶段流转。
- 前端在不同设备、不同分辨率下美观合理，动画过渡自然，平台稳定可靠，内容动态呈现。

#### 1.3 适用范围

本平台覆盖创实部职责范围内的典型工作，包括但不限于：“挑战杯”系列竞赛、“互联网+”大赛、大学生创新创业训练计划（大创）、科研立项、寒暑期社会实践、共青团重点调研课题申报，以及其他学科竞赛的通知传达、人员统计与费用收集。后续可扩展至其他院级竞赛与事务。

***

### 二、用户角色与职责

> 角色权限按以下顺序依次递减：部长 > 副部长 > 干事 > 游客。

| 角色             | 职责与权限概述                                                               |
| -------------- | --------------------------------------------------------------------- |
| **部长**（最高权限）   | 统筹所有项目，监控进度，协调资源，最终审核并发送汇总表。可分配副部长至具体项目。对全平台数据可读、可改；管理全部账号（创建、重置密码）。                  |
| **副部长**        | 负责特定比赛项目的执行。在项目内划分阶段任务，指定干事负责每个团队的每项材料，审核干事提交的材料，生成阶段汇总表。仅可操作本人负责的项目。 |
| **干事**         | 执行具体材料的收集工作。查看自己名下的待办任务，向参赛团队收集指定文件并上传，标记完成。仅可操作分配给自己的任务。             |
| **游客**（只读浏览） | 未登录访客，仅可浏览公开的项目/阶段/材料提交状态，**无权新增、修改或删除任何内容**，亦无权上传材料；材料收取以干事代收为主。  |

### 三、核心业务流程

#### 3.1 总体流程

1. **通知下达与项目创建**\
   学校发布比赛通知 → 部长/副部长在平台创建对应比赛项目（可基于模板），设置各阶段时间节点。
2. **团队信息入库**\
   副部或干事将报名的团队/个人基本信息录入系统（支持批量导入），形成参赛名单。
3. **材料任务指派**\
   副部在项目某阶段中，为每个团队的每项待交材料指定一名负责干事，系统自动生成待办任务。
4. **干事收件与上传**\
   干事查看自己的任务清单，联系对应团队收取材料，将电子版上传至平台，标记为“已提交”。
5. **副部审核**\
   副部逐项审核材料完整性、规范性，通过后标记“已审核”；若有问题可打回重新提交。
6. **汇总表生成与报送**\
   本阶段所有团队材料审核完成后，副部点击“生成汇总表”，系统自动拉取团队信息与提交状态，导出标准 Excel 表格，连同电子材料提交学院团委盖章并报送学校。
7. **阶段流转**\
   当前阶段结束后，项目进入下一阶段（如从中期检查进入结题），系统可按预设自动生成下一阶段所需任务草案，供副部确认并指派干事。
8. **答辩/评审组织（可选阶段）**\
   需答辩的阶段，副部可在系统内安排答辩时间、地点，录入评审结果。

#### 3.2 涉及多阶段项目的特殊处理

- 平台需明确区分项目下的不同阶段，每个阶段独立管理材料任务。
- 阶段可以有依赖关系（如必须先完成中期检查才能开启结题），也可手动开启。
- 同一比赛项目的各阶段可关联同一批参赛团队，避免重复录入。

### 四、功能性需求

#### 4.1 基础数据管理

**4.1.1 比赛模板管理**

- 支持创建比赛模板，包含：模板名称（如“大创项目”）、预设阶段清单。
- 每个阶段可预设：阶段名称、顺序、建议时长、需收集的材料清单（材料名称、说明）、是否需答辩/评审。
- 可编辑、复制、停用模板。

**4.1.2 人员管理**

- 维护部门成员信息：姓名、学号、角色（部长/副部长/干事）、联系方式。
- 支持批量导入。
- 系统根据角色自动授予数据权限。

**4.1.3 账号与登录**

- 仅部长、副部长、干事拥有登录账号；游客无需登录即可浏览公开内容。
- **部长账号固定**：用户名 `admin`，初始密码 `12345678`，不可删除、不可由他人重置。
- **副部、干事账号由部长创建**：部长预设成员姓名，成员初次登录时从已预设的姓名列表中选择自己的名字，并设置密码完成激活。
- **登录交互**：登录页输入姓名时提供**模糊匹配的下拉候选框**，选中后输入密码即可登录。
- **密码安全**：密码以哈希形式存储，任何人（含部长）均不可查看他人密码；部长可对任一成员执行**密码格式化（重置）**，重置后该成员需重新设置密码。
- **忘记密码**：成员线下向部长反馈，由部长在后台重置其密码。

#### 4.2 项目管理

**4.2.1 创建项目**

- 选择比赛模板快速创建项目实例，自动继承模板的阶段和材料清单设置。
- 填写项目基本信息：项目名称、比赛级别、年份、相关通知附件。
- 指定 1-2 名副部长作为项目负责人。
- 设置各阶段的起止时间（可覆盖模板默认值）。
- 部长默认为所有项目的最高监管人，不在此处指派。

**4.2.2 项目总览（部长驾驶舱）**

- 以卡片或列表形式展示所有进行中的项目，显示：项目名称、当前阶段、整体进度（已完成任务数/总任务数）、负责人。
- 逾期预警：高亮显示超过截止日期仍未完成的任务数量，支持一键查看详情。
- 人员负荷视图：可视化展示每位副部长负责的项目数，每位干事手头的任务数，便于协调。

**4.2.3 项目详情与监控**

- 部长和项目负责副部长可进入项目查看：所有阶段、所有团队、所有材料任务的状态。
- 支持按阶段、按团队、按干事筛选。
- 可手动发送催办提醒给相关干事（如生成通知消息）。

#### 4.3 参赛团队管理

- 可在项目内创建、编辑、删除参赛团队（或个人）信息。
- 信息字段：团队名称、队长姓名、学号、联系方式、成员名单（可多行文本或关联子表）、指导老师。
- 支持从 Excel 批量导入团队数据。
- 支持在项目各阶段间自动同步团队名单，也可灵活增删（如中期有团队退出）。

#### 4.4 阶段与任务管理（核心）

**4.4.1 阶段控制**

- 一个项目包含若干有序阶段，列表显示：阶段名、状态（未开始/进行中/已完成）、起止时间。
- 副部长可手动切换阶段状态，但系统应校验是否所有任务已完成。
- 支持“启动下一阶段”操作，系统询问是否根据模板自动生成该阶段的材料任务（可选复用上一阶段团队名单）。

**4.4.2 材料任务创建与指派**

- 在一个阶段内，针对每个参赛团队，需要收集多类材料（如申报书、诚信承诺书、汇总表等）。
- 系统可批量生成任务：选择阶段、选择团队范围、选择材料清单项，为每队每项材料生成一个“材料任务”。
- 副部长在每个材料任务上指派一名干事负责人。支持批量指派（如将所有团队的“申报书”任务全部指派给干事A）。
- 任务包含字段：所属项目、阶段、团队、需交材料名称、截止日期、负责干事、提交状态（待提交/已提交/已审核/已打回）、文件附件、备注。

**4.4.3 干事工作台**

- 干事登录后，首页显示“我的任务”列表，按截止时间排序。
- 每条任务清晰显示：比赛项目-阶段-团队-材料名称-截止时间-状态。
- 干事上传文件后，可将状态从“待提交”更改为“已提交”，并可添加备注。
- 支持查看任务历史，已逾期任务用红色标注。
- 如果允许参赛学生直传，可提供表单链接，学生提交后自动匹配到对应任务并通知干事，但此处以干事上传为主要模式。

#### 4.5 材料审核与汇总

**4.5.1 副部审核视图**

- 项目负责副部长可查看本项目下所有干事提交的材料（可按阶段、干事、状态筛选）。
- 预览或下载文件，确认后标记为“已审核”；若有缺失或错误，可“打回”并填写原因，任务状态退回“待提交”，干事需重新上传。
- 支持批量审核通过。

**4.5.2 一键生成汇总表**

- 在任一阶段，副部可点击“导出阶段汇总表”。
- 系统根据预设的模板（如学校要求的 Excel 格式）自动生成表格，包含：序号、团队名称、队长、成员、联系方式、材料1提交状态、材料2提交状态……审核备注等。
- 导出前可预览，生成真实的 .xlsx 文件下载。
- 部长同样拥有导出所有项目汇总表的权限。

**4.5.3 报送管理**

- 副部在完成线下盖章报送后，可在系统内标记该阶段“已报送”，并上传盖章扫描件存档。
- 阶段完结后，该项目整体进度自动更新。

#### 4.6 通知与提醒

- 任务逾期前 1 天，系统自动向负责干事推送提醒（如邮件、飞书/钉钉消息，视平台能力）。
- 当干事提交材料后，副部可收到通知。
- 部长可手动对特定项目、阶段下的所有逾期任务发送批量催办。
- 当项目阶段即将截止时，提醒副部长。

#### 4.7 答辩与评审管理（扩展模块）

- 对于需要答辩的阶段，副部长可创建“答辩安排”。
- 录入答辩时间、地点、评委信息。
- 为每个参赛团队分配答辩顺序、时间。
- 录入每个团队的评审结果（分数、等级、是否通过）。
- 评审结果可体现在汇总表或单独导出。

### 五、数据存储设计

考虑到服务器内存有限、无专职运维人员，本平台**不使用任何数据库服务**（不部署 MySQL/PostgreSQL 等常驻进程），改用**基于文件系统的 JSON 文件存储**方案。数据以结构化 JSON 文件落盘，按项目隔离存放，读写由轻量后端直接操作文件完成，内存占用极低。

#### 5.1 存储结构

采用\*\*「一项目一目录」的沙箱式隔离\*\*：每个比赛项目在服务器上对应一个独立、自包含的目录，目录名直接使用「项目名\_年份」（如 `大创项目_2026`），所见即所得。整个项目目录可直接打包导出，无需额外整理。

根目录建议设在 `/var/www/zhansai/data`：

```
/data
  /templates/                 比赛模板（每个模板一个 .json 文件）
    {模板名称}.json
  /users.json                 全部部门成员（单文件，数组）
  /projects.json              全部项目索引（单文件，数组，记录每个项目的目录名）
  /projects/
    大创项目_2026/             ← 一个项目 = 一个自包含目录（沙箱）
      meta.json               项目基本信息
      stages.json             该项目所有阶段（数组）
      teams.json              该项目所有参赛团队（数组）
      tasks.json              该项目所有材料任务（数组）
      /uploads/               该项目全部材料附件
        申报阶段/张三队/申报书.pdf      用「阶段名/团队名/文件名」分层，人眼可读
        申报阶段/李四队/申报书.pdf
      /exports/               生成的汇总表（.xlsx）存档
```

> **目录名规则**：取「项目名\_年份」并过滤掉 `/ \ : * ? " < > |` 等文件系统非法字符；若重名则追加序号（如 `大创项目_2026_2`）。
>
> **一键导出**：对任意项目执行 `tar -czf 大创项目_2026.tar.gz /data/projects/大创项目_2026/`，即可得到包含全部数据、附件与汇总表的完整压缩包，可直接归档或移交学校。

#### 5.2 实体字段定义

各 JSON 文件内记录的字段如下（字段设计与原数据模型一致，仅由“表”改为“文件中的对象”）。

**5.2.1 比赛模板（templates/{模板ID}.json）**

- 模板ID、模板名称
- 阶段列表（数组：阶段顺序、阶段名称、建议时长、需收集材料清单、是否答辩）

**5.2.2 项目（projects/{项目目录名}/meta.json + projects.json 索引）**

- 项目目录名（如 `大创项目_2026`）、项目ID、项目名称、使用模板ID、年份
- 状态（进行中/已完结）
- 负责副部长ID（数组）
- 创建时间

**5.2.3 项目阶段（projects/{项目目录名}/stages.json）**

- 阶段ID、所属项目ID、阶段名称、顺序号
- 开始日期、截止日期、是否答辩
- 状态（未开始/进行中/已完成）
- 材料清单（如 \["申报书", "汇总表", "诚信承诺书"]）

**5.2.4 参赛团队（projects/{项目目录名}/teams.json）**

- 团队ID、所属项目ID、团队名称（同时作为 uploads 子目录名）
- 队长姓名、学号、联系方式
- 成员信息（文本）、指导老师、备注

**5.2.5 材料任务（projects/{项目目录名}/tasks.json）**

- 任务ID、所属阶段ID、所属团队ID
- 材料名称、负责干事ID、截止日期
- 提交状态（待提交/已提交/已审核/已打回）
- 文件附件（路径形如 `uploads/{阶段名}/{团队名}/{原始文件名}`，人眼可读）
- 审核意见、创建时间、最后更新时间

**5.2.6 人员（users.json）**

- 用户ID、姓名、学号
- 角色（部长/副部/干事）
- 联系方式
- 账号信息：密码哈希、是否已激活（初次登录设置密码标记）、是否为固定账号（admin）

#### 5.3 读写与并发说明

- 所有写操作由后端统一串行处理：写入前读取文件、修改后整文件覆盖写回，并先写临时文件再原子重命名（rename），防止中途异常写坏数据。
- 单服务器、低并发场景下无需复杂锁机制；如需更强一致性，可对单个项目文件加文件锁。
- JSON 文件即天然备份：可配合 crontab 每日打包 `/data` 目录实现增量备份，也可直接拷贝到对象存储。

**关系简述**：一个项目目录下包含多个阶段、多个团队、多个材料任务；每个任务属于一个团队并指定一名干事。副部长通过项目 meta 中的“负责副部长ID”关联，部长角色拥有全局查看权限。

### 六、非功能性需求

- **易用性**：界面简洁，操作流程贴合原有工作习惯，干事端任务列表清晰，无需大量培训。
- **响应式与多端适配**：前端在不同设备（桌面、平板、手机）、不同分辨率下均能美观、合理地展示与操作，动画过渡自然流畅。
- **性能与稳定性**：平台面向约 13-15 人日常使用，需保证稳定运行、响应迅速；后端部署于轻量化服务器，资源占用低。
- **权限控制**：严格按角色划分数据权限。干事只能看到自己被指派的任务；副部长只能操作自己负责的项目；部长可查看所有，但修改需谨慎设计。
- **数据安全**：材料文件落盘存储并定期备份（亦可上传对象存储）。敏感信息（如学号）传输加密；密码哈希存储，不可逆向查看。
- **可扩展性**：比赛模板和阶段可灵活调整，支持未来新增比赛类型与事务（如费用收集、人员统计）。
- **低维护成本**：优先采用轻量自建（JSON 文件存储 + 极简后端），无数据库等常驻服务，运维负担小；后端工作文件按第五部分目录结构合理规划。

### 七、实现建议（非功能性需求补充）

鉴于服务器内存有限且希望数据自主可控，提供两条落地路径，按优先级排列。

#### 7.1 方案一（首选）：轻量自建，JSON 文件存储 + 静态前端

在自有服务器上以极低内存方式自建，**全程不使用数据库服务**，数据以「一项目一目录」的沙箱形式落盘，便于打包导出与归档。

- **前端**：纯静态页面（HTML/CSS/JS 或 Vue 单页），由 Nginx 托管，内存占用近乎为零。
- **后端**：Python Flask 或 Node.js Express，仅提供 JSON 文件的读写与附件上传接口，常驻内存约 30-60MB。
- **数据存储**：按第五部分的 JSON 文件结构落盘，无数据库进程；每个项目独立目录，可直接 `tar` 打包整目录导出。
- **汇总表导出**：后端用 openpyxl（Python）或 exceljs（Node）读取 tasks.json 生成 .xlsx，存入该项目的 `exports/` 目录。
- **部署**：Nginx + Flask/Express + systemd，无需 MySQL/Redis 等服务，整机内存占用可控在 100MB 以内。
- **备份**：crontab 每日打包 `/data` 目录至 `/backup` 或对象存储。

该方案数据完全自主可控、对服务器内存友好，且项目目录即天然归档包，是当前推荐路径。
